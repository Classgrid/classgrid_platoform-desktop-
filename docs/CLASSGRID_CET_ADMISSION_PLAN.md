### Key Principles:
- **EN Number = Identity** (from CET, source of truth)
- **Email = Verification** (Verified Personal Email at start via OTP)

### 🎟️ THE POST-CAP ENGINE (SPOT vs INST vs MGMT)
| Feature | SPOT ROUND | INSTITUTIONAL SEATS | MANAGEMENT QUOTA |
|---|---|---|---|
| **Origin** | Post-CAP Vacancies | College Reserve Quota | Private Seats |
| **EN Number** | **MANDATORY** | **MANDATORY** | **OPTIONAL** |
| **Merit Basis** | CET Percentile | CET Percentile | Flexible (Admin Choice) |
| **Scholarship** | Category-Based | **Full Fees Only** | **Full Fees Only** |

**Logic Enforcement:**
1. **SPOT/INST:** Backend requires EN + Percentile. Builds automated Merit Lists.
2. **MANAGEMENT:** Bypasses EN check. Direct Admin selection.
3. **FEE LOCK:** Institutional and Management paths automatically disable scholarship discounts.

- **Credential Bridge:** Once enrolled, the **New College Email** (Username) and **User-set Password** are sent to the **Verified Personal Email**.
- **OTP = Security** (Ensures credentials go to the correct owner)
- ERP is the processing system, NOT the data provider
- CET remains the source of truth for allotment

## 🏛️ Engineering Has THREE Admission Modes

| Mode | Entry Point | Who | Auth Method |
|---|---|---|---|
| **1. Apply from Home** | Student/Parent fills online from home | Quota students (Management/NRI/Minority) — NO EN | Phone OTP (Firebase) |
| **2. Fill at College Lab** | Walk-in → Admin sends to lab → Fill same online form | Quota students — NO EN | Phone OTP (Firebase) |
| **3. Fill at College Lab with EN** | Walk-in CET student → Admin sends to lab → Enter EN code | CET-allotted students (CAP seats) | EN → Email OTP |

> **School / Junior College / Coaching** = Only Mode 1 + 2 (no EN code exists).
> **Engineering** = All 3 modes (Mode 1 + 2 for Quota, Mode 3 for CET).

### Phase B: Student Authentication (Public)

```javascript
// EN Checksum Validation (MOD-11 Algorithm)
export function validateENNumber(en_number, institute_code) {
  en_number = en_number.trim().toUpperCase();
  const pattern = /^EN(\d{2})(\d{5})(\d{4,5})(\d{1})$/;
  const match = en_number.match(pattern);
  if (!match) return { valid: false, error: 'Invalid EN format' };
  
  const [, year, college, serial, checksum] = match;
  const currentYear = new Date().getFullYear() % 100;
  if (parseInt(year) !== currentYear && parseInt(year) !== currentYear - 1) {
    return { valid: false, error: `EN year ${year} not valid` };
  }
  if (college !== institute_code) {
    return { valid: false, error: `College code mismatch` };
  }
  const digits = en_number.slice(2, -1).split('').map(Number);
  let sum = 0;
  for (let i = 0; i < digits.length; i++) {
    sum += digits[i] * (digits.length - i);
  }
  if (parseInt(checksum) !== (sum % 11)) {
    return { valid: false, error: 'EN checksum validation failed' };
  }
  return { valid: true, data: { year, college, serial, checksum } };
}
```

## 🔄 CAP Round Upgrade Handling (Real DTE Flow)

When a student gets a better college in a later CAP round (e.g., CAP-II):

```
Student was allotted College A in CAP-I. 
They reported, paid fees, and selected "Float" (Want better college).

CAP-II Allotment Published:
Student gets College B.

1. Student must physically go to College A.
2. Parent signs "Upgrade Consent Form".
3. College A issues NOC (No Objection Certificate) and releases student in DTE portal.
4. College A marks student's record as 'upgraded_out' in our system.
5. College A's seat becomes vacant (triggers waitlist or next round filling).
6. Student goes to College B to report and verify documents.
7. College B imports CAP-II PDF → Student's EN appears → Starts fresh admission.
```

**System Requirements for Upgrade:**
- Prevent "double counting": Same EN cannot be `enrolled` in two colleges.
- NOC tracking is mandatory for releasing a student.
